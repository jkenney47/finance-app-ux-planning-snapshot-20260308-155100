# Gemini Plan Review

- Plan: `docs/plans/2026-03-03-figma-xcode-ui-workflow.md`
- Generated-At-UTC: 2026-03-04T01:48:50Z
- Model: gemini-3.1-pro-preview

Here is the critique of the **UI Workflow (Figma Sync + Xcode Validation)** implementation plan:

### 1. P0/P1/P2 Risks
*   **P1 - Velocity Impact & Developer Friction:** Mandating strict Xcode validation (Dynamic Type, VoiceOver, Landscape, multiple simulator sizes) and Figma synchronization for *every* minor UI change (e.g., a simple color tweak or padding adjustment) will severely bottleneck development. The policy needs a threshold (e.g., "New Components" vs. "Minor Tweaks").
*   **P2 - Enforcement Failure:** Relying solely on documentation and templates means this process relies purely on discipline. Without automated checks (e.g., a script parsing the PR for checked boxes), it is highly likely to be bypassed.

### 2. Missing Steps or Hidden Dependencies
*   **Target Paths Omission:** `AGENT_TASK_TEMPLATE.md` is mentioned at the top of the plan but missing from the "Target Paths". If agents are expected to summarize validation using this template, it must also be updated to include the new Figma/Xcode checklist requirements.
*   **Enforcement Script:** The plan lacks a step to create or update a `.git/hooks/pre-push` or GitHub Action (e.g., `scripts/check-pr-ui-checklist.sh`) to verify the checklist exists and is completed in the PR body.

### 3. Frontend State Coverage (loading/error/empty/offline)
*   *Process scope only, but...* The workflow document (`docs/ux/ui-development-workflow.md`) and the `agent-spec.md` template MUST explicitly mandate that Figma designs and Xcode validations account for Loading, Error, Empty, and Offline states, not just the "happy path".

### 4. Accessibility Risks
*   *Process scope only, but...* While VoiceOver and Dynamic Type are correctly included in the Xcode validation checklist, **Color Contrast / Minimum Hit Targets (44x44pt on iOS)** are missing from the explicit validation list and should be added.

### 5. Concrete Tests to Add or Run
*   **Dry Run:** Test the new process by drafting a dummy `agent-spec.md` for a small UI change to verify the checklist isn't overly burdensome.
*   **Tooling Validation:** Ensure that running `npm run validate` (mentioned in the test plan) actually captures the presence of the new template sections.

### 6. Simpler Alternatives
*   **Automate over Manual:** Instead of relying entirely on manual Xcode simulator validation for every state, rely on **Maestro UI tests** for structural/functional regressions, and reserve manual VoiceOver/Dynamic Type simulator testing strictly for *new* components or major refactors. 
*   **Tiered Checklist:** Create a "Light" checklist for minor style tweaks and a "Full" checklist for new views to balance quality with speed.
