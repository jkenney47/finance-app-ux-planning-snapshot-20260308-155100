MCP issues detected. Run /mcp list for status.Here is the critique of the proposed execution plan:

### 1. P0/P1/P2 Risks
*   **P1 (Backend Coupling Risk):** The plan states "No Supabase function changes", but if the backend's `create_link_token` or `exchange_public_token` RPCs implicitly rely on the `ENABLE_REAL_ACCOUNT_DATA` flag being true, the frontend decoupling will immediately crash in production/staging. Verify the backend does not tightly couple these flags before execution.
*   **P1 (UX Confusion):** A user successfully completes the real Plaid Link flow (in Sandbox) but is routed to a dashboard that shows completely unrelated mock data because `ENABLE_REAL_ACCOUNT_DATA=false`. This disconnected experience will confuse testers. You must handle the handoff gracefully (e.g., a toast: "Sandbox linked. Dashboard remains in mock mode").
*   **P2 (Environment Leakage):** Ensure that enabling `ENABLE_PLAID_SANDBOX_LINK` without `ENABLE_REAL_ACCOUNT_DATA` strictly enforces the `sandbox` Plaid environment in `utils/account.ts` and never defaults to `development` or `production`.

### 2. Missing Steps & Hidden Dependencies
*   **Post-Link Routing:** The plan mentions adding a "safe linked-account verification path" but misses the explicit routing step. After verification, where does the user go? 
*   **Mock State Collision:** If `useDashboardData.ts` continues to return mock data, does the app state still track the newly linked sandbox account `itemId` or `institutionName`? The local state shape must cleanly support having a "real" linked account in memory while the dashboard renders mock mode.

### 3. Frontend State Coverage (UI Involved)
*   **Verification Loading State:** If a new verification step is added post-exchange, `plaid-link.tsx` needs a distinct loading state (e.g., "Verifying account...").
*   **Verification Error State:** What happens if the public token exchange succeeds, but the new "safe verification path" fails? The UI must handle this partial failure without leaving the user in a dead end.
*   **Offline Mode:** If the user drops connection between the Plaid modal closing and the verification request firing, the error must be caught and gracefully surfaced.

### 4. Accessibility Risks
*   If the "safe linked-account verification path" introduces new visual indicators (loading spinners or success checks) before routing, ensure they are announced to screen readers (e.g., `aria-live="polite"`).
*   Ensure focus is managed properly after the external Plaid modal closes and returns control to the React Native app.

### 5. Concrete Tests to Add/Run
*   **`__tests__/utils/account.test.ts`:** Add a matrix test proving `getPlaidLinkToken` and `exchangePlaidPublicToken` succeed when `SANDBOX_LINK=true` and `REAL_ACCOUNT_DATA=false`.
*   **`__tests__/app/(auth)/plaid-link.test.tsx` (Missing in plan):** Add a component test for `plaid-link.tsx` asserting the UI shows the new verification loading state and routes correctly on success when `REAL_ACCOUNT_DATA=false`.
*   **E2E (Maestro):** Run a Maestro flow that specifically tests the sandbox link success path while the dashboard mock flag is active to ensure the handoff doesn't crash the app.

### 6. Simpler Alternatives
*   **Skip Frontend Verification:** The plan proposes adding a "safe linked-account verification path". Is this strictly necessary for Phase A2? If the goal is just to decouple the flags so the Link UI can open, you could simply exchange the token, log the success, and immediately route to the dashboard. Deferring the complex verification logic to Phase A3 (when real dashboard data is actually consumed) significantly reduces the complexity and risk of this current slice.
